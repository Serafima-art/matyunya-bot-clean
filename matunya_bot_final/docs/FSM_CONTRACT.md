📜 FSM_CONTRACT_v2.md

Контракт состояний (FSM) проекта «Матюня»
Архитектура: non-generators

🎯 Назначение

Этот документ описывает минимальные правила работы FSM
для заданий, построенных по схеме:

router → loader → solver → humanizer → send

FSM в новой архитектуре:

❌ не является источником истины

✅ отвечает только за навигацию и UX

✅ хранит текущий контекст пользователя

1️⃣ Источник истины
📌 Источник истины = конкретный task

Solver всегда получает:

{
    "variant": task_payload,
    "task": task
}

Где:

variant — весь вариант (таблица, данные, метаданные)

task — конкретный вопрос (Q1–Q5)

FSM не определяет, какой вопрос решается.
FSM лишь хранит текущий выбор пользователя.

2️⃣ Обязательные поля FSM

Для заданий 1–5 FSM обязан содержать:

task_type = "1-5"
task_subtype = "paper" | "stoves" | ...
task_1_5_data = {...}        # загруженный вариант
current_task_index = int     # 0-based
current_question_number = int  # 1-based
3️⃣ Главный инвариант
❗ При вызове Help должен существовать:
state["task_1_5_data"]

и должна быть возможность определить конкретный task
по question_num.

Если это невозможно:

помощь не запускается

логируется ERROR

пользователю отправляется сообщение об ошибке

4️⃣ Роль FSM в новой архитектуре

FSM отвечает только за:

хранение текущего выбранного вопроса

контроль состояния waiting_for_answer

хранение:

solution_core

help_image

help_opened

solved_tasks_indices

FSM не отвечает за:

вычисление индексов

определение вопроса внутри solver

восстановление "потерянных" индексов

логику бизнес-вычислений

5️⃣ Запрещено

⛔ Нельзя:

рассчитывать номер вопроса внутри solver

передавать только index без task

делать solver зависимым от FSM

восстанавливать вопрос из случайных state-полей

6️⃣ Router — единственная точка выбора

Выбор вопроса происходит только в:

dispatch_focused_screen

Именно router:

получает question_num

сохраняет его в FSM

передаёт правильный task в solver

7️⃣ Упрощение по сравнению с legacy

Удалено:

state["index"]

ensure_task_index()

fallback-логика

восстановление question_num

FSM как «якорь логики»

FSM больше не является контрактом вычислений.
Он — контракт навигации.

8️⃣ Расширение на будущие задания

Этот контракт обязателен для:

заданий 1–5 (non-generators)

заданий 16+

всех будущих pattern → narrative → facts архитектур

Если задание использует:

solver получает task

значит действует FSM_CONTRACT_v2.

9️⃣ Философия новой модели

Проект развивается в сторону:

чистых solver'ов

отсутствия скрытой магии

отсутствия зависимостей от FSM

явной передачи контекста

FSM — это UI-слой.
Solver — это логика.
Humanizer — это представление.

Никаких пересечений ролей.
