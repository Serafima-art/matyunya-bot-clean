🧭 Механизм маршрутизации заданий 1–5 (новая архитектура 01.03.2026)
🎯 Общая идея

Система заданий 1–5 построена по принципу:

Router = диспетчер
Registry = список специалистов
Subtype = конкретный специалист

Router не знает деталей реализации подтипов.
Он только передаёт управление нужному модулю.

🧠 Участники системы
1️⃣ task_1_5_router.py

Главный вход в систему.

Отвечает за:
обработку callback
управление state
регистрацию задач в БД
очистку сообщений
отображение загрузки и ошибок
навигацию (карусель, возврат, выбор вопроса)

Router не знает, как устроены Paper, Stoves и другие подтипы.

2️⃣ registry.py

Файл-конфигурация.

Содержит словарь соответствия:

TASK_1_5_REGISTRY = {
    "paper": {
        "loader": ...,
        "overview": ...,
        "focused": ...,
    },
    "stoves": {
        "loader": ...,
        "overview": ...,
        "focused": ...,
    },
}

Registry сообщает Router’у:

«Если subtype = paper, вот кто за него отвечает».

Registry не работает с Telegram и не содержит бизнес-логики.

3️⃣ Подтип (subtypes/<name>/)

Каждый подтип — изолированный модуль.

Структура:

subtypes/
    paper/
        loader.py
        overview.py
        focused.py

Подтип полностью отвечает за:
выбор варианта
формирование display_scenario
построение интро
отрисовку обзорного экрана
отрисовку фокусного задания

🔁 Полный сценарий работы
Шаг 1 — выбор подтипа

Пользователь выбирает, например:

1–5 → Бумага

Router получает:

subtype_key = "paper"
Шаг 2 — поиск специалиста

Router обращается к registry:

handler = TASK_1_5_REGISTRY[subtype_key]
Шаг 3 — загрузка варианта

Router вызывает:

task_1_5_data = await handler["loader"](...)

Loader:
выбирает случайный вариант
строит интро
прикрепляет изображения
формирует display_scenario
возвращает готовый словарь данных

Router не знает, как это реализовано.

Шаг 4 — регистрация задач

Router:
регистрирует задачи в БД
сохраняет correct_answers
обновляет state

Шаг 5 — отображение экрана

Router вызывает:
await handler["overview"](...)
или
await handler["focused"](...)
🚫 Что запрещено в Router

В Router не должно быть:

if subtype == "paper":
if subtype == "stoves":

Router не знает конкретных подтипов.

➕ Добавление нового подтипа

Чтобы добавить, например, apartment:

Создать папку:
subtypes/apartment/
Реализовать:
loader.py
overview.py
focused.py

Добавить одну запись в registry.py:
"apartment": {...}
Router менять не требуется.

🏗 Архитектурный принцип

Router — управляет процессом
Registry — хранит соответствия
Subtype — реализует логику

Каждый слой отвечает только за свою зону ответственности.

Это позволяет:
масштабировать систему до 10–12 подтипов
добавлять новые темы без изменения Router
избежать каскадных правок
поддерживать чистую архитектуру
